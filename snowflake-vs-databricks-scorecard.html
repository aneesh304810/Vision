🗄 SQL & Relational Compatibility
ANSI SQL Compliance HIGH
Full ANSI SQL. Oracle migration path is clean — most PL/SQL logic translates directly. dbt-snowflake is best-in-class.
SQL support is strong but Databricks leans Spark SQL — subtle dialect differences. More friction migrating Oracle stored procedures.
SF
Stored Procedures & PL/SQL Migration HIGH
Snowflake Scripting is close to PL/SQL. Snowpark lets you lift complex Oracle procs into Python/Java with familiar patterns.
Databricks requires rewrite to PySpark or Delta Live Tables — significantly more effort for a legacy Oracle shop.
SF
dbt Compatibility HIGH
dbt-snowflake is the most mature, best-documented adapter. First-class support, every dbt feature works out of the box.
dbt-databricks works but Delta-specific materializations add complexity. Some features lag behind dbt-snowflake support.
SF
🏛 Regulatory & Compliance (NYDFS · SOC2 · Data Residency)
Data Residency Controls HIGH
Single-region account isolation. Private Link native. Data never leaves designated region. NYDFS-friendly by default.
Possible but requires careful Unity Catalog configuration. Multi-cloud nature adds complexity to proving data residency.
SF
Audit & Query History HIGH
ACCOUNT_USAGE and INFORMATION_SCHEMA — full query history, access logs, login events out of the box. NYDFS audit trail ready.
Audit logs available but require Databricks Unity Catalog + Azure Monitor integration. More setup to get equivalent coverage.
SF
Row & Column Level Security MED
Native row access policies and column masking — critical for wealth management client data isolation.
Unity Catalog supports this but configuration is more complex and Unity Catalog adoption maturity is lower.
SF
⚙ Infrastructure & Operations
Zero Infrastructure Management HIGH
True serverless — no clusters to manage, no Spark tuning, no executor config. Spin up a VW, run SQL, auto-suspend.
Databricks clusters require sizing, auto-scaling config, Spark tuning. Less approachable for a SQL-first banking team.
SF
Learning Curve for SQL Team HIGH
SQL-first. A banker or analyst who knows Oracle can use Snowflake in days. No Spark, no JVM, no cluster concepts.
Spark paradigm requires significant upskilling for a team coming from Oracle/SQL Server. PySpark is not SQL.
SF
Cost Predictability MED
Per-second VW billing with auto-suspend is highly predictable. Separate compute from storage clearly.
DBU (Databricks Units) pricing is harder to forecast. Cluster sizing mistakes can cause bill spikes.
SF
🏗 Data Architecture Fit
Structured / Tabular Data HIGH
Purpose-built for structured relational data. Perfect fit for positions, transactions, NAV, AUM — the core of wealth management.
Overkill for pure structured data. Databricks shines with semi-structured and unstructured — not BBH's primary use case today.
SF
Time Travel & Data Recovery MED
Snowflake Time Travel (90 days) + Fail-safe. Point-in-time recovery built in.
Delta Lake also has time travel via transaction log. Comparable capability.
TIE
ML / AI Workloads LOW for now
Snowflake Cortex and Snowpark ML are growing but less mature than Databricks for training large models.
Databricks is the clear winner for ML/AI — MLflow native, GPU cluster support, best-in-class for model training and feature engineering.
DB

Possible but not Snowflake's strength. VARIANT type handles JSON but not images, PDFs natively.
Databricks handles unstructured data natively — PDFs, emails, documents. Better for future AI use cases on client documents.

SEI SWP Integration (current program) HIGH
SEI's platform already runs on Snowflake. BBH VW slots directly into existing SEI account. Zero integration work.
Would require a parallel Databricks environment alongside SEI's Snowflake — two platforms, two teams, higher cost and complexity.
SF
Oracle Migration Tooling HIGH
Snowflake's ecosystem has mature Oracle migration tools — schema conversion, data type mapping, SQL translation utilities.
Fewer Oracle-specific migration tools. Delta migration from Oracle requires more custom engineering.
SF
Marketplace / Partner Ecosystem LOW
Snowflake Marketplace — data sharing, third-party datasets. Strong for financial data vendors.
Databricks Marketplace growing but smaller. Both have strong partner ecosystems for banking.
TIE

❄ Snowflake — Choose When

  • Primary workload is structured relational data from Oracle
  • Team is SQL-first — no Spark expertise or appetite to build it
  • NYDFS audit trail and data residency are non-negotiable
  • SEI SWP is already on Snowflake — align to existing platform
  • Need zero infrastructure management — banking IT ops is lean
  • dbt is the transformation standard — best adapter is Snowflake
  • Cost predictability matters for budget governance

⚡ Databricks — Choose When

  • BBH invests seriously in ML/AI — model training on client data
  • Unstructured data processing becomes a core use case (PDFs, emails)
  • You need a unified platform for both data engineering AND data science
  • Streaming / real-time event processing becomes a requirement
  • BBH builds its own AI models vs consuming models from vendors
  • Long-term — pair with Snowflake, don't replace it
VERDICT Snowflake
Clear Win
For a legacy bank migrating from Oracle with structured relational data, NYDFS regulatory requirements, and a SQL-first engineering team — Snowflake wins on every dimension that matters today. The SEI SWP program already runs on Snowflake, making it the path of zero friction. Databricks is not a bad product — it is the wrong product for this specific context and maturity stage. The right long-term answer is not Snowflake vs Databricks — it is Snowflake as the warehouse layer with Databricks introduced later as the AI/ML layer when BBH's data science capability matures, running against the same Delta-compatible storage if needed.
