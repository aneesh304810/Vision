The Use Case Only Airflow Can Handle

Dynamic DAG generation + XCom-driven branching + sensor-gated execution + runtime task map expansion — all in a single, observable, recoverable pipeline.

The Problem Space

⚡ Custody Batch: Adaptive Multi-Fund Settlement with Dynamic Task Mapping

A financial services pipeline that ingests an unknown number of fund portfolios at runtime, runs per-fund position reconciliation in parallel, dynamically branches per instrument type, waits on external settlement signals, cross-contaminates state via XCom, and self-heals with partial re-run capability — all within a 4-hour SLA window.

🗺️

Dynamic Task Mapping

Number of funds is unknown until a sensor reads the settlement file at runtime. Airflow 2.3+ .expand() creates N tasks dynamically — no pre-coded fan-out.

🔀

XCom-Driven Branching

Each fund task pushes instrument metadata into XCom. A downstream BranchPythonOperator reads cross-task XComs to route equities, fixed income, and derivatives into different sub-pipelines.

👁️

External Signal Sensors

SWIFT MT940 settlement confirmation is polled via a custom ExternalTaskSensor before funds close. Poke mode, timeout, and exponential backoff are all independently configured per fund.

🔄

Partial Re-run / Task-level Recovery

When 3 of 47 funds fail settlement validation, only those 3 task instances are cleared and retried — not the entire DAG run. No other orchestrator does this without full job re-submission.

🕸️

Cross-DAG Dependency

The downstream NAV calculation DAG is triggered only after all mapped settlement tasks succeed, via ExternalTaskSensor with execution_date_fn remapping for business calendar shifts.

📊

Unified Observability

Every task instance — including dynamically generated ones — appears in the Airflow UI Grid View with per-instance logs, duration metrics, and retry history. Zero external tooling needed.

.expand() creates N reconcile tasks at runtime

Airflow's Dynamic Task Mapping API creates one reconcile_fund task instance per fund ID at DAG parse time of the run — not at authoring time. No Jinja loops, no pre-generated DAGs, no factory patterns needed.

Python · Airflow 2.5+ 
# ── Dynamic Task Mapping: fan-out across unknown fund count ──

with DAG("custody_settlement", schedule="0 18 * * 1-5",
         catchup=False, max_active_runs=1) as dag:

    @task
    def get_fund_ids() -> list[dict]:
        # Reads manifest from S3 — N is unknown at authoring time
        manifest = s3.read_json("s3://settlement/manifest.json")
        return manifest["funds"]   # e.g. [{"id":"F01","ccy":"USD"}, ...]

    @task
    def reconcile_fund(fund: dict) -> dict:
        # Runs once per fund — in parallel, up to pool limit
        result = run_position_recon(fund["id"])
        return {"fund_id": fund["id"], "instrument_type": result.type,
                "breaks": result.breaks}

    funds    = get_fund_ids()
    recon    = reconcile_fund.expand(fund=funds)  # ← THE MAGIC LINE
    # Airflow creates N task instances, visible in Grid View

Python · XCom + BranchPythonOperator 
@task.branch
def route_by_instrument(recon_results: list[dict]) -> list[str]:
    """
    Reads XCom from ALL mapped reconcile tasks via the injected list.
    Returns which downstream task IDs to activate — others are SKIPPED.
    This is a conditional fan-in/fan-out: only active instrument
    types proceed, avoiding unnecessary compute on quiet days.
    """
    types = {r["instrument_type"] for r in recon_results}
    active = []
    if "EQUITY"       in types: active.append("equity.settle_equities")
    if "FIXED_INCOME" in types: active.append("fi.settle_bonds")
    if "DERIVATIVE"   in types: active.append("deriv.settle_otc")
    return active   # multiple branches can be active simultaneously

# Wire: recon results flow into the branch via XCom automatically
router = route_by_instrument(recon_results=recon)

with TaskGroup("equity") as equity_group:
    settle_equities = PythonOperator(task_id="settle_equities", ...)

with TaskGroup("fi") as fi_group:
    settle_bonds = PythonOperator(task_id="settle_bonds", ...)

router >> [equity_group, fi_group, deriv_group]
The XCom fan-in trick: When recon_results is declared as a parameter of the downstream task, Airflow automatically collects XComs from ALL mapped task instances (all N fund reconciliations) and passes them as a single aggregated list. This cross-task state aggregation with automatic fan-in is unique to Airflow's task decorator model.
Python · Custom Sensor with Exponential Backoff 
class SwiftMT940Sensor(BaseSensorOperator):
    """
    Polls SWIFT BIC endpoint for MT940 settlement confirmation.
    - poke_interval: starts at 60s, backs off to 300s max
    - timeout: 14400s (4hr SLA window)
    - mode='reschedule': releases worker slot between pokes
      (critical for not starving the Celery worker pool)
    - soft_fail=True: marks skipped rather than failed
      when no settlement expected (e.g. holiday calendar)
    """

    def __init__(self, fund_id: str, expected_ccy: str, **kwargs):
        super().__init__(
            mode="reschedule",          # ← non-blocking: release slot between pokes
            timeout=14400,
            poke_interval=60,
            exponential_backoff=True,   # ← Airflow 2.6+ native
            max_wait=300,
            **kwargs
        )
        self.fund_id = fund_id
        self.expected_ccy = expected_ccy

    def poke(self, context: Context) -> bool:
        confirmed = swift_api.check_mt940(
            fund=self.fund_id,
            value_date=context["logical_date"].date(),
            ccy=self.expected_ccy
        )
        if confirmed:
            # Push confirmation ref into XCom for audit trail
            context["task_instance"].xcom_push(
                key="swift_ref", value=confirmed.reference
            )
        return confirmed is not None

# One sensor created per currency group (also dynamically mapped)
swift_gate = SwiftMT940Sensor.partial(task_id="swift_mt940_confirm")

3 of 47 fund reconciliations fail at 21:45

The 3 mapped task instances for reconcile_fund[12], reconcile_fund[28], reconcile_fund[41] go to FAILED state. All other 44 instances complete successfully and downstream processing for those funds continues.

Task-level isolation
2

Ops clears only the 3 failed instances from the UI

From the Grid View, an operator selects the 3 failed mapped instances and clicks "Clear". Airflow re-runs only those 3 tasks — the root cause (a stale position feed) is fixed upstream and the retry succeeds within 8 minutes.

Selective clear + re-run
3

DAG run completes — SLA preserved

The 3 recovered tasks push their XCom results, the downstream route_by_instrument task is re-evaluated with the full 47-fund result set, and NAV calculation DAG is triggered. Total recovery time: 11 minutes. No data re-ingestion, no full re-run.

SLA metZero re-ingestion
The killer differentiator: In Temporal, Prefect, or Luigi, a "re-run" means re-submitting the entire workflow or re-playing from a checkpoint. Airflow's metadata database holds every task instance state independently. You can surgically clear and re-execute any subset of tasks, in any run, including dynamically-mapped instances — while keeping successful neighbors intact.
