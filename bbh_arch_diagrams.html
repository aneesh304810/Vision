Architecture Diagrams & Data Flow

Option 1: Full Azure Stack  |  Option 2: Hybrid BBH OpenShift + Snowflake
DATA FLOW STEPS — OPTION 1
STEP 1 · INGESTION

SEI SWP generates account / portfolio files and drops them to the BBH-managed SFTP server. Files are CSV / pipe-delimited.

STEP 2 · AZURE BLOB LANDING

Azure Blob Storage picks up files from SFTP and lands them in the raw landing container. Key Vault supplies Snowflake + SFTP credentials to all services.

STEP 3 · AIRFLOW ORCHESTRATION

Airflow 3.x on AKS detects new files in Blob, triggers downstream DAGs, manages retries and SLAs. DAG configs also stored in Blob.

STEP 4 · SNOWFLAKE STAGE LOAD

Blob files are loaded via COPY INTO to Snowflake Internal Stage. No transformation yet — raw data preserved for auditability.

STEP 5 · dbt TRANSFORMATION

dbt-snowflake pods on AKS execute Bronze → Silver → Gold SQL models. Compute runs inside Snowflake Virtual Warehouses — AKS pod only sends SQL.

STEP 6 · CONSUMER ACCESS

~1000 downstream consumers query Snowflake Gold marts directly via JDBC/ODBC. BI tools, APIs, reports, and client-facing apps all connect to Snowflake endpoint.

DATA FLOW STEPS — OPTION 2
STEP 1 · INGESTION

SEI SWP drops files to the BBH-managed SFTP server. Same SFTP pattern as today — no change to SEI side.

STEP 2 · AIRFLOW PICKUP

Airflow 3.x on OpenShift polls SFTP via FileSensor. Existing DAG infrastructure, Jenkins CI/CD, Splunk observability — all unchanged.

STEP 3 · dbt-SNOWFLAKE TRANSFORM

dbt-snowflake pods (KubernetesPodOperator) execute Bronze → Silver → Gold SQL. Only change: adapter swap from dbt-oracle to dbt-snowflake + profiles.yml update. SQL models largely reusable.

STEP 4 · HTTPS BOUNDARY CROSSING

dbt pushes SQL over HTTPS :443 outbound from BBH OpenShift to Snowflake on Azure. This is the key data residency boundary — legal/Hema must clear.

STEP 5 · SNOWFLAKE LAYERS

Bronze → Silver → Gold transformation executes inside Snowflake Virtual Warehouses. OpenShift pod only sends SQL — data never leaves Snowflake back to BBH.

STEP 6 · ORACLE PARALLEL RUN

Strangler fig pattern continues. Oracle + Advent AddVantage stays live. Consumers migrated incrementally via dbt compatibility layer views. Oracle decommissioned last.

STEP 7 · CONSUMER ACCESS

New consumers query Snowflake Gold directly. Legacy consumers remain on Oracle compat layer until cutover. Dual-feed during migration window.
